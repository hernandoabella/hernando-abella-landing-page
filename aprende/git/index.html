Tutorial de GIT

Introducción

Git es un Sistema de Control de Versiones muy popular. Fue creado por Linus Torvalds en 2005 y desde entonces ha sido mantenido por Junio ​​Hamano.

Se utiliza para:

  • Hacer seguimiento de cambios en el código
  • Hacer seguimiento de quién hizo los cambios
  • Colaborar en la codificación

¿Qué hace Git?

  • Manejar proyectos con Repositorios
  • Clonar un proyecto para trabajar en una copia local
  • Controlar y rastraer los cambios con Staging y Commiting
  • Utilizar Branch y Merge te permite trabajar en diferentes partes y versiones de un proyecto.
  • Pull (Extraiga) la última versión del proyecto a una copia local
  • Push (envía) actualizaciones locales al proyecto principal.

Trabajando con Git

  • Inicializa Git en una carpeta haciéndola un Repositorio
  • Ahora Git crea una carpeta oculta para mantener el rastro de los cambios en esa carpeta
  • Cuando un archivo es cambiado, agregado o eliminado, se considera modificado
  • Seleccionas los archivos modificados que quieras ordenar
  • Los archivos en etapas están confirmados, lo que le dice a Git que almacene una captura de pantalla de los archivos.
  • Git te permite ver un historial completo de cada commit
  • Puedes revertir cualquier commit anterior
  • Git no almacena una copia separada de cada archivo en cada commit, pero realiza un seguimiento de los cambios realizados en cada commit.

¿Por qué Git?

  • ¡Más del 70% de los desarrolladores usan Git!
  • Los desarrolladores pueden trabajar juntos desde cualquier parte del mundo.
  • Los desarrolladores pueden ver el historial completo del proyecto.
  • Los desarrolladores pueden volver a versiones anteriores de un proyecto.

¿Qué es GitHub?

  • Git no es lo mismo que GitHub.
  • GitHub crea herramientas que usan Git.
  • GitHub es el mayor host de código fuente del mundo y ha sido propiedad de Microsoft desde 2018.
  • En este tutorial, nos centraremos en usar Git con GitHub.

Empezando

Instalar Git

Puede descargar Git de forma gratuita desde el siguiente sitio web: https://www.git-scm.com/

Usando Git con la línea de comandos

Para comenzar a usar Git, primero vamos a abrir nuestro shell de comandos.

Para Windows, puede usar Git bash, que viene incluido en Git para Windows. Para Mac y Linux, puede utilizar el terminal integrado.


Lo primero que debemos hacer es verificar si Git está instalado correctamente:

git --version

Resultado:

git version 2.30.2.windows.1

Configurar Git

Ahora hazle saber a Git quién eres. Esto es importante para los sistemas de control de versiones, ya que cada commit de Git utiliza esta información:

git config --global user.name "Hernando Abella"
git config --global user.email "hernandoabella@gmail.com"

Cambie el nombre de usuario y la dirección de correo electrónico por los suyos. Probablemente también desee usar esto cuando se registre en GitHub más adelante.

Nota: Usa --global para ajustar el nombre de usuario y el correo electrónico de cada repositorio en tu computadora.


Si quieres ajustar el nombre de usuario / correo electrónico para el repositorio actual, puedes remover global.

Creando una carpeta Git

Ahora crearemos una carpeta nueva para nuestro proyecto:

mkdir miproyecto
cd miproyecto

mkdir crea un nuevo directorio.


cd cambia la carpeta de trabajo actual.


Ahora que estamos en la carpetaa correcta. ¡Podemos inicializar Git!

Nota: Si ya tienes una carpeta para Git:


Navega en la línea de comandos, o ábrela desde el explorador de archivos, click-derecho y selecciona "Git Bash here"

Inicializar Git

Una vez que hayas navegado a la carpeta correcta, puedes inicializar Git en esa carpeta:

Ejemplo:

git init

Resultado:

Initialized empty Git repository in /Users/user/myproject/.git/


¡Haz creado tu primer Respositorio!

Nota: Ahora Git sabe que debe observar la carpeta que haz inicializado.


Git crea una carpeta oculta para llevar el seguimiento de los cambios.

Nuevos archivos de Git

Agregando nuevos archivos

Acabas de crear tu primer repositorio local de Git. Pero está vacio.


Así que agreguemos algunos archivos, o creemos un archivo nuevo usando nuestro editor de texto favorito. Luego guárdalo y muévelo a la carpeta que acabas de crear.


Para este ejemplo, voy a utilizar un archivo HTML simple como este:

<!DOCTYPE html>
<html>
<head>
<title>¡Hola Mundo!</title>
</head>
<body>

<h1>¡Hola Mundo!</h1>
<p>Este es el primer archivo en mi nuevo Repo Git.</p>

</body>
</html>

Y guárdelo en nuestra nueva carpeta como index.html.


Regresemos a la terminal y enumeremos los archivos en nuestra carpeta de trabajo actual:

ls

Resultado:

index.html


ls enlistará los archivos de la carpeta. Podemos ver que index.html está ahí.


Luego verificamos el status de Git y vemos si es parte de nuestro repositorio:

git status

Resultado:

On branch master

No commits yet

Untracked files:
  (use "git add ..." to include in what will be committed)
    index.html

nothing added to commit but untracked files present (use "git add" to track)

Ahora Git conoce el archivo, ¡pero no lo ha agregado a nuestro repositorio!


Los archivos en su carpeta de repositorio de Git pueden estar en uno de 2 estados:


Tracked: archivos que Git conoce y que se agregan al repositorio


Untracked: archivos que están en su directorio de trabajo, pero no agregados al repositorio.


Cuando agrega archivos por primera vez a un repositorio vacío, no se realiza el seguimiento de todos ellos. Para que Git los rastree, debe organizarlos o agregarlos a el área de preparación.

Área de preparación

Una de las funciones principales de Git son los conceptos del área de preparación y el commit.


Mientras trabaja, puede agregar, editar y eliminar archivos. Pero cada vez que alcanza un hito o termina una parte del trabajo, debe agregar los archivos a un área de preparación.


Los archivos preparados son archivos que están listos para ser enviados al repositorio en el que estás trabajando.


Por ahora, hemos terminado de trabajar con index.html. Entonces podemos agregarlo a el área de preparación.

git add index.html

El archivo debería estar preparado. Verifiquemos su estado:

git status

Resultado:

On branch master

No commits yet

Changes to be committed:
  (use "git rm --cached ..." to unstage)
    new file: index.html

Ahora el archivo ha sido agregado al área de preparación.

Agrega más de un archivo

También puedes preparar más de un archivo al mismo tiempo. Agreguemos 2 archivos más a nuestra carpeta. Usa el editor de texto nuevamente.


Un archivo README.md describe el repositorio (recomendado para todos los repositorios):

# Hola Mundo

Tutorial del repositorio de Hello World para Git

Este es un repositorio de ejemplo para el tutorial de Git en https://www.hernandoabella.com

Este repositorio se crea paso a paso en el tutorial.

Una hoja de estilo externa básica (bluestyle.css):

body {
    background-color: lightblue;
}

h1 {
    color: navy;
    margin-left: 20px;
}

Ahora agrega todos los archivos de la carpeta actual al área de preparación:

git add -all

El uso de --all en lugar del nombre de los archivos individuales preparará todos los cambios de los archivos (nuevos, modificados y eliminados).

git status

Resultado:

On branch master

No commits yet

Changes to be committed:
  (use "git rm --cached ..." to unstage)
        new file:   README.md
        new file:   bluestyle.css
        new file:   index.html

Ahora los 3 archivos han sido agregados al área de preparación y estamos listo para hacer nuestro primer commit.

Nota: El comando abreviado para git add --all es git add -A

Git Commit

Agregar commits realiza un seguimiento de nuestro progreso y cambios a medida ue trabajamos. Git considera cada punto de guardado. Es un punto del proyecto al que puedes volver si encuentras un error o si deseas realizar un cambio.


Cuando hacemos commit, siempre debemos incluir un mensaje.


Al agregar menajes claros a cada confirmación, es fácil para ti (y para los demás) ver qué ha cambiado y cuándo.

git commit -m "¡Primera versión de Hola Mundo!"

Resultado:

[master (root-commit) 221ec6e] ¡Primera versión de Hola Mundo!
 3 files changed, 26 insertions(+)
 create mode 100644 README.md
 create mode 100644 bluestyle.css
 create mode 100644 index.html

El commando commit realiza una confirmación y el -m "mensaje" agrega un mensaje.


El área de preparación ha sido enviada a nuestro repositorio con el mensaje:


"¡Primera versión de Hola Mundo!"

Commit sin preparación

Aveces, cuando realizas pequeños cambios, utilizar el área de preparación parece una pérdida de tiempo. Es posible realizar cambios directamente, omitiendo el área de preparación. la opción -a organizará automáticamente todos los archivos modificados y ya registrados.

Agreguemos una pequeña actualización a index.html:

<!DOCTYPE html>
<html>
<head>
<title>Hello World!</title>
<link rel="stylesheet" href="bluestyle.css">
</head>
<body>

<h1>¡Hola Mundo!</h1>
<p>Este es el primer archivo en mi nuevo Git Repo.</p>
<p>¡Una nueva línea en nuestro archivo!</p>

</body>
</html>

Verifique el estado de nuestro repositorio. Pero esta vez, usaremos la opción --short para ver los cambios de una manera más compacta:

git status --short

Resultado:

M index.html

Nota: Los indicadores de estado cortos son:

  • ?? - Archivos sin seguimiento
  • A - Archivos agregados al área de preparación
  • M - Archivos modificados
  • D - Archivos eliminados

Vemos que el archivo que esperábamos está modificado. Así que vamos a actualizarlo directamente

git commit -a -m "index.html actualizado con una nueva línea"

Resultado:

[master 09f4acd] Updated index.html with a new line
 1 file changed, 1 insertion(+)

Advertencia: No se recomienda omitir el área de preparación.


Saltarse el paso de preparación puede hacer que incluya cambios no deseados.

Para ver el historial de commits de un repositorio, puedes usar el comando log:

Ramas

Trabajando con Ramas

En Git, un branch (rama) es una versión nueva / separada del repositorio principal.


Digamos que tienes un proyecto extenso, y necesitas actualizar su diseño.

Cómo funcionaría con o sin Git:

Sin Git:

  • Hace copias de todos los archivos relevantes para evitar afectar la versión en vivo
  • Empiece a trabajar con el diseño y descubra que el código depende del código de otros archivos, ¡que también deben cambiarse!
  • Haga también copias de los archivos dependientes. Asegúrese de que cada dependencia de archivo haga referencia al nombre de arcivo correcto.
  • ¡Emergencia! Hay un error no relacionado en algún otro lugar del proyecto que debe arreglarse lo antes posible.
  • Guarde todos sus archivos, tomando nota de los nombres de las copias en las que estaba trabajando.
  • Trabaje en el error no relacionado y actualice el código para solucionarlo
  • Regrese al diseño y termine el trabajo allí.
  • Copie el código o cambie el nombre de los archivos, para que el diseño actualizado esté en la versión en vivo.
  • (2 semanas después, se da cuenta de que el error no relacionado no se solucionó en la nueva versión de diseño porque copió los archivos antes de la corrección)

Con Git:

  • Con una nueva rama llamada nuevo diseño, edite el código directamente sin afectar la rama principal
  • ¡Emergencia! hay un error no relacionado en algún otro lugar del proyecto que debe arreglarse lo antes posible
  • Cree una nueva rama del proyecto principal llamada small-error-fix
  • Corrija el error no relacionado y combine la rama de corrección de errores pequeños con la rama principal
  • Vuelve a la rama de nuevo diseño y termina el trabajo allí.
  • Fusiona la rama del nuevo diseño con la rama principal (main) recibiendo una alerta sobre la pequeña corrección de error que faltaba

Las ramas te permiten trabajar en diferentes partes de tu proyecto sin tener ningun impacto con la rama principal.


Cuando el trabajo está completo, una rama puede ser fusionada con el proyecto principal.


Incluso puedes cambiar entre ramas y trabajar en diferentes proyectos sin que interfieran entre sí.

La ramificación en Git es muy rápida y ligera.

Nueva Rama de Git

Agreguemos algunas características nuevas a nuestra pagina index.html


Estamos trabajando en nuestro repositorio local y no queremos molestar o posiblemente destruir el proyecto principal.


Asi que creamos una nueva rama:

git branch imagenes-hola-mundo

Ahora hemos creado una nueva rama llamada "imagens-hola-mundo"


Confirmemos que ha sido creada una nueva rama:

git branch

Resultado:

  imagenes-hola-mundo
* master

Podemos ver la nueva rama con el nombre de "imagenes-hola-mundo", pero el asterisco * al lado de master especifica que actualmente estamos en esa rama.


El comando checkout se usa para verificar una rama. Moviéndonos de la rama actual, a la especificada al final del comando:
